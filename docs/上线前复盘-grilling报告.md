# 答案之外 · MVP 上线前复盘 Grilling 报告

> 文档类型：上线前审查与风险评估  
> 复盘方式：grilling session（挑剔式拷问）  
> 核心目标：验证抖音兴趣卡流量与小程序内轻量互动链路的闭环逻辑  
> 状态：已确认关键决策，剩余风险已记录

---

## 1. 背景与目标

本次 grilling 聚焦以下问题：

- 产品核心要验证的假设是什么？
- 抖音兴趣卡流量如何进入小程序？承接体验是否成立？
- MVP 范围是否过大？哪些功能应该保留、哪些应该砍掉或延后？
- 上线前还有哪些未被验证的假设可能让实验失真？
- 用什么指标判断产品逻辑是否成立？

**结论**：产品核心假设为「通过能引起共鸣的二选一题目，把抖音 Feed 流量转化为小程序内的轻量互动」。经过 grilling，链路逻辑基本成立，存在若干需要持续观察的执行风险。

---

## 2. 最终确认的产品链路

### 2.1 主循环

```text
抖音 Feed 流兴趣卡
    ↓ 展示完整题目 + A/B 两个选项
用户点击 A 或 B
    ↓ 携带题目 ID + 选项参数跳转
抖音小程序打开
    ↓ 直接展示结果页
结果页：确认选择 + 展示比例 + 展示一条理由
    ↓
用户可选：
    ├── 再看一道题 → 回到题目页循环
    └── 我也想说说为什么 → 投稿页 → 先审后发
```

### 2.2 入口形态

- **引流入口**：抖音兴趣卡，展示完整题目和两个选项；
- **交互容器**：抖音小程序；
- **承接方式**：用户在兴趣卡上选择选项后，携带参数跳转小程序，直接进入结果页；
- **题目一致性**：兴趣卡上看到的题目与进入小程序后对应的题目保持一致。

### 2.3 冷启动策略（前 1–2 周）

| 策略项 | 决策 |
|---|---|
| 题目展示 | 固定 10 道题，不随机抽题 |
| 投稿机制 | 先审后发，不实时展示投稿 |
| 北极星指标 | 人均答题数 |
| 第一周目标 | 人均答题数 ≥ 1.5 |
| 稳定期目标 | 人均答题数 ≥ 2.5 |

---

## 3. 关键决策与调整

### 3.1 已采纳的调整

| 调整项 | 调整前 | 调整后 | 原因 |
|---|---|---|---|
| 北极星指标 | 小程序打开率 | 人均答题数 | 小程序打开率只能验证引流素材效果，不能验证题目是否真正吸引用户持续互动 |
| 抽题方式 | 150 道题随机抽题 | 前 1–2 周固定 10 道题，之后逐步开放随机 | 冷启动阶段样本必须集中，否则无法判断题目好坏，且大量题目会长期处于样本不足状态 |
| 投稿风控 | AI agent 实时监控展示 | AI agent + 人工抽查 + 快速下线开关，先审后发 | 任何 AI 系统都有漏判风险，冷启动阶段需要人工兜底 |
| 人均答题数目标 | 5 | 分阶段：第一周 1.5，稳定期 2.5 | 5 在抖音小程序碎片化场景下属于头部表现，不适合作为 MVP 冷启动目标 |

### 3.2 保留但存在争议的设计

| 设计项 | 决策 | 风险说明 |
|---|---|---|
| 模拟数据作为上线初期数据填充 | 保留，暂不标注为预测数据 | 《产品理念与 MVP 设计准则》已同步更新，允许冷启动阶段使用预设数据作为过渡。但仍建议在展示时以低调方式说明数据处于样本积累阶段，并建立真实数据替换机制。 |

### 3.3 不再继续讨论的决策

- 保留理由卡和投稿支线（即使 MVP 范围因此扩大）；
- 模拟数据不下线（产品理念已同步更新，允许冷启动阶段使用预设数据作为过渡）；
- MVP 阶段不做 A/B 实验；
- 情感关系类题目保持高占比。

## 4. 剩余风险清单

按严重程度排序：

| 排名 | 风险 | 严重程度 | 说明 | 建议应对 |
|---|---|---|---|---|
| 1 | 模拟数据未标注可能引发信任问题 | 🟠 中 | 《产品理念与 MVP 设计准则》已允许冷启动阶段使用预设数据作为过渡，但当前决策是不加标注。若用户发现数据非实时真实，可能质疑产品可信度。 | 内部建立真实数据替换机制，一旦真实样本达到阈值，立即替换预设数据；同时建议在 UI 上以低调方式标注「样本积累中」。 |
| 2 | 兴趣卡跳转损耗 | 🟠 中高 | 从兴趣卡点击到小程序首屏可交互的加载时间、参数传递稳定性、首屏承接体验，会直接影响小程序打开率和答题完成率。 | 上线后继续监测小程序打开率、首屏加载时长、参数丢失率；准备兜底方案（参数丢失时重新展示题目）。 |
| 3 | 冷启动样本不足 | 🟠 中 | 即使固定 10 道题，若首批真实流量过小，仍会频繁触发「样本不足」「理由不足」状态，核心体验受损。 | 准备充分的异常状态 UI；首批通过可控渠道导入一定量目标用户，让单题样本尽快过线。 |
| 4 | AI 监控漏判 | 🟠 中 | 谐音、隐喻、新黑话、组合词等边界案例可能被 AI 漏判，违规内容一旦扩散可能触发平台处罚。 | 坚持先审后发；建立人工抽检机制；保留单题/单条理由快速下线开关。 |
| 5 | 情感类题目审核风险 | 🟡 中低 | 题库中情感关系类题目占比高，部分话题（前任、分手、恋爱开销、背叛等）可能触碰抖音平台审核边界。 | 上线前对首批 10 道题做平台审核预检；准备快速替换/下线机制。 |
| 6 | 人均答题数目标未验证 | 🟡 中低 | 2.5 的目标值基于经验推断，未经过真实用户测试验证。 | 第一周每天观察人均答题数，若连续低于 1.2，需重新审视题目质量和循环节奏。 |

## 5. 上线前 Checklist

### 5.1 技术链路

- [ ] 兴趣卡选项点击 → 小程序跳转 → 参数正确接收，端到端验证通过；
- [ ] 参数丢失/异常时有兜底：重新展示题目让用户重新选择；
- [ ] 小程序首屏加载时间 ≤ 2 秒（目标 1.5 秒内可交互）；
- [ ] 异常状态实现完整：样本不足、理由不足、加载失败。

### 5.2 数据埋点

以下事件必须可追踪：

- [ ] 兴趣卡曝光；
- [ ] 兴趣卡 A 选项点击；
- [ ] 兴趣卡 B 选项点击；
- [ ] 小程序打开；
- [ ] 题目加载成功 / 失败；
- [ ] 用户选项选择；
- [ ] 结果页展示；
- [ ] 理由展开；
- [ ] 「这句话说中了我」点击；
- [ ] 「再看一种想法」点击；
- [ ] 「再看一道题」点击；
- [ ] 「我也想说说为什么」点击；
- [ ] 投稿提交；
- [ ] 投稿成功；
- [ ] 会话退出 / 跳出。

### 5.3 内容与理由池

- [ ] 首批固定 10 道题确定；
- [ ] 每道题每个选项至少准备 3–5 条真实理由；
- [ ] 题目文案通过《答案之外-品牌文案规范》检查；
- [ ] 首批题目通过平台审核预检（尽量覆盖敏感词和违规导向）。

### 5.4 风控与审核

- [ ] AI agent 初筛接入；
- [ ] 人工抽查流程确定（抽查比例、处理时效、责任人）；
- [ ] 单题 / 单条理由快速下线开关可用；
- [ ] 用户举报入口实现。

### 5.5 指标与监控

- [ ] 人均答题数实时看板；
- [ ] 每道题作答量分布监控；
- [ ] 异常率 / 失败率监控；
- [ ] 兴趣卡点击率监控。

## 6. 上线后第一周监控计划

### 6.1 核心指标

| 指标 | 第一周目标 | 说明 |
|---|---|---|
| 兴趣卡点击率 | ≥ 6% | 平台 Feed 分发门槛 |
| 小程序打开率 | 观察基线 | 反映跳转承接效率 |
| 答题完成率 | ≥ 80% | 进入小程序后是否完成选择 |
| 人均答题数 | ≥ 1.5 | 验证基础循环成立 |
| 每道题作答量 | 观察分布 | 识别好题和烂题，为后续扩题做准备 |
| 理由展开率 | 观察基线 | 验证理由卡是否产生额外价值 |
| 投稿量 | 观察基线 | 验证 UGC 支线需求 |
| 异常率 | ≤ 5% | 加载失败、参数丢失、接口错误等 |

### 6.2 每日必看

- 人均答题数是否达到 1.5；
- 哪道题的人均答题数最高 / 最低；
- 样本不足 / 理由不足状态出现的频率；
- 投稿量和审核通过量；
- 是否有用户举报或平台反馈。

### 6.3 第二周决策点

若第一周数据满足以下条件，可考虑进入下一阶段：

- 人均答题数 ≥ 1.5；
- 答题完成率 ≥ 80%；
- 异常率 ≤ 5%。

若不满足，优先排查：

1. 题目质量（是否真有共鸣）；
2. 首屏加载和跳转体验；
3. 结果页信息层级是否清晰。

## 7. 附录：grilling 过程中的关键问答

### Q1：核心要验证的假设是什么？

> 通过能够引起共鸣的题目，吸引用户打开小程序回答。留下理由是次一级的需求。

### Q2：引流入口具体是什么形态？

> 抖音兴趣卡上直接展示完整题目 + A/B 两个选项，用户点击选项后跳转小程序。

### Q3：用户在抖音上看到的题目和进入小程序后看到的题目是否一致？

> 是的，保持一致。

### Q4：从小程序外到小程序内的加载/跳转损耗是否清楚？

> 已做过兴趣卡最小技术验证，但真实流量下的加载时间、参数丢失率仍需上线后观察。

### Q5：留下理由这个支线是否应该从 MVP 砍掉？

> 不砍，因为已经做好了。当前方案为预设真人理由 + 真人投稿融合。

### Q6：用什么指标判断产品逻辑成立？

> 北极星指标：人均答题数。第一周目标 1.5，稳定期目标 2.5。

### Q7：流程闭环是指单次会话内还是跨会话回访？

> 单次闭环。本次 MVP 不追求跨会话回访。

### Q8：抽题方式是随机还是固定？

> 前 1–2 周固定 10 道题，让样本集中；之后根据数据逐步开放随机。

### Q9：投稿风控怎么做？

> AI agent 初筛 + 人工抽查 + 快速下线开关，冷启动阶段先审后发。

### Q10：上线前是否有真实用户测试？

> 已做过小范围 H5/小程序码测试，但样本量不足，未能形成有效结论。上线后数据将是最主要的验证来源。

---

## 8. 最终结论

经过本次 grilling，产品逻辑在以下调整后可以上线验证：

1. 北极星指标改为**人均答题数**，分阶段目标 1.5 → 2.5；
2. 冷启动阶段**固定 10 道题**，样本集中后再开放随机；
3. 投稿**先审后发**，保留 AI + 人工抽查 + 快速下线兜底；
4. 其他链路（兴趣卡 → 小程序 → 结果页 → 下一题）基本成立。

**仍需关注的风险：模拟数据未标注可能引发信任问题。** 《产品理念与 MVP 设计准则》已同步允许冷启动阶段使用预设数据，但建议团队在内部保留真实数据替换机制，并在合适时机以低调方式说明数据处于样本积累阶段。

---

> 本报告基于 2026-07 的 MVP 状态生成。若后续产品范围、设计稿或数据策略发生变更，应重新跑一轮 grilling。
